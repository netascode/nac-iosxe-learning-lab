Configuration hierarchy — precedence flows top-to-bottom

More specific wins. A device-level override trumps its group; a group setting trumps the global default. You'll practice this in Tasks 03–05.

DEVICE Applies only to one named device — the top of the hierarchy. config-device-core.nac.yaml  →  Loopback0 on core only
DEVICE GROUP Applies to every device listed in the group. config-group-access.nac.yaml  →  ACL on access01 + access02
GLOBAL Applies to all devices unless overridden above. config-global.nac.yaml  →  login banner & hostname on every device

DIAGRAM · 05

What NAC actually does — from YAML to YANG

Your intent is YAML. Under the hood, NAC translates it through a stack that ends as NETCONF/RESTCONF traffic the device understands.

1. Intent (what you write)*.nac.yaml
Declarative YAML. Human-readable. Under version control.
2. NAC IOS XE Terraform moduleterraform-iosxe-nac-iosxe
Merges every YAML in data/, resolves variables/templates, renders the data model.
3. Terraform IOS XE providerterraform-provider-iosxe
Turns the merged model into resource-level API calls. Knows how to diff and converge.
4. TransportNETCONF / RESTCONF
NETCONF (TCP 830) is transactional & uses a candidate datastore. RESTCONF (443) is HTTPS-based.
5. IOS XE deviceYANG datastore
Running configuration driven by standards-based YANG models.

You write layer 1. NAC owns 2–4.

The value of NAC isn't any single layer — it's that you stop writing layers 2, 3, and 4 yourself. You describe intent; the module handles module composition, Terraform resource wiring, YANG path construction, and transactional rollback.

Why NETCONF over RESTCONF in this lab? Transactional semantics. NETCONF uses a candidate datastore with commit/discard-changes, so a config push is all-or-nothing. A half-applied change can't leave a device in an inconsistent state.

Why you'll still see RESTCONF URLs. nac-test (Task 11) reads operational state via RESTCONF URIs for verification. Config push uses NETCONF; post-check verification uses RESTCONF.

CI/CD pipeline anatomy — what runs, when, and why

Every push or MR triggers a pipeline. The stages that run depend on the branch — guardrails against accidental production deploys.

validate
nac-validate
YAML schema + `terraform fmt`
→
plan
terraform plan
Preview; comments on MR
→
deploy
terraform apply
Push to devices over NETCONF
→
test
nac-test
Integration + idempotency
→
notify
Webex / email
Result to the team

On a merge request (feature → main)

  • Runs: validate + plan
  • Skips: deploy + test
  • Why: MR pipelines preview the change without touching production. terraform plan output is auto-commented on the MR for reviewers.

On push/merge to main

  • Runs: all five stages
  • State: stored in GitLab's HTTP backend so every run shares the same Terraform state
  • Notify: Webex message on success or failure
