What NaC actually does — from YAML to YANG

Your intent is YAML. Under the hood, NaC translates it through a stack that ends as NETCONF/RESTCONF traffic the device understands.

1. Intent (what you write)*.nac.yaml
Declarative YAML. Human-readable. Under version control.
2. NaC IOS XE Terraform moduleterraform-iosxe-nac-iosxe
Merges every YAML in data/, resolves variables/templates, renders the data model.
3. Terraform IOS XE providerterraform-provider-iosxe
Turns the merged model into resource-level API calls. Knows how to diff and converge.
4. TransportNETCONF / RESTCONF
NETCONF (TCP 830) is transactional & uses a candidate datastore. RESTCONF (443) is HTTPS-based.
5. IOS XE deviceYANG datastore
Running configuration driven by standards-based YANG models.

You write layer 1. NaC owns 2–4.

The value of NaC isn't any single layer — it's that you stop writing layers 2, 3, and 4 yourself. You describe intent; the module handles module composition, Terraform resource wiring, YANG path construction, and transactional rollback.

Why NETCONF over RESTCONF in this lab? Transactional semantics. NETCONF uses a candidate datastore with commit/discard-changes, so a config push is all-or-nothing. A half-applied change can't leave a device in an inconsistent state.

Why you'll still see RESTCONF URLs. nac-test (Task 11) reads operational state via RESTCONF URIs for verification. Config push uses NETCONF; post-check verification uses RESTCONF.

NETCONF datastores — why NaC uses it instead of RESTCONF

NETCONF separates "the change you're describing" from "the config actually on the device" — so a half-applied change can't leave a device broken.

CANDIDATE DATASTORE Proposed configuration Writable, private scratch space. No effect on traffic.
<edit-config>Terraform writes here
<validate>Device checks syntax + semantic
<commit> ✓Atomic swap → candidate becomes running
<discard-changes> ✗Throw the candidate away, device untouched
RUNNING DATASTORE Live configuration on the device What traffic is actually forwarded against right now.

The "transactional" part

Any single edit-config + commit cycle is atomic at the device level. Either every change you described takes effect together, or none of it does.

That means if Terraform is pushing a 40-line configuration change and line 37 is invalid, the device's running config stays exactly as it was — not in a half-applied broken state.

RESTCONF doesn't work this way

RESTCONF writes directly to the running datastore, one HTTP call per resource. No candidate, no commit, no rollback primitive. A mid-batch failure leaves the device in whatever partial state the successful calls produced.

For a lab this rarely matters. For a production push across 100 devices? It matters a lot — which is why NaC defaults to NETCONF when it's available, and why this lab does too.

YAML merge semantics — three rules, one outcome

NaC merges every data/*.nac.yaml file using three deterministic rules. Understanding them is the difference between "my config deployed" and "why is there a duplicate device entry?"

RULE 1 · DICT Dicts deep-merge by key Keys from both documents are combined. If the same key appears in both, values are merged recursively.

hostname: r1

hostname: r1
domain: lab
RULE 2 · LIST OF PRIMITIVES Lists of scalars are appended Strings, numbers, bools — concatenated. Order preserved. Duplicates NOT deduped.

- 10.0.0.1
- 10.0.0.2
+

- 10.0.0.3
ntp_servers:
- 10.0.0.1

- 10.0.0.3
