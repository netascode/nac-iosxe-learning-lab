In this task, you'll learn how to use **templates of type 'file'** to reference external configuration files. The `file` template type allows you to store large or complex configurations in separate files and reference them from your device YAML, keeping your main configuration organized and manageable.

## Understanding 'file' Templates

Unlike `model` templates that contain YAML-based configuration directly, `file` templates reference external files that contain the actual configuration. This is particularly useful for:

- **Large configurations**: When configuration blocks become too large to manage inline
- **External sources**: When configurations are generated by other tools or systems
- **Separation of concerns**: When you want to keep complex configurations in dedicated files
- **Legacy integration**: When importing existing configuration files into Network-as-Code

**Template Types (reminder):**

| Type | Description | Use Case |
|------|-------------|----------|
| `model` | YAML-based configuration template | Standard configurations (VLANs, ACLs, etc.) - *Task06* |
| `file` | External file reference | Large configurations stored separately - *This task* |
| `cli` | Raw CLI commands | Legacy or complex CLI configurations - *Task08* |

## Use Case: BGP Configuration from External File

In this example, you'll configure BGP peering using a `file` template. The BGP configuration will be stored in a separate file and referenced from your device YAML.

!!! info "Pre-configured Neighbor"
    The BGP neighbor switch is already configured with BGP in this lab. When you apply this configuration, the BGP session should establish automatically.

## Step 1: Create the BGP Configuration File

First, create a new file `data/templates/bgp_core.txt` containing the BGP configuration:

```text
router bgp 65001
 bgp router-id interface Loopback0
 bgp log-neighbor-changes
 neighbor 198.18.100.2 remote-as 65002
```

This file contains:

- **AS 65001**: Your local autonomous system number
- **Router-ID from Loopback0**: Uses the Loopback0 IP as the BGP router ID
- **Neighbor 198.18.100.2**: The eBGP peer in AS 65002

## Step 2: Create the Template Reference

Create a new file `data/bgp_template.nac.yaml` with the following content:

```yaml
templates:
  - name: bgp_core_config
    type: file
    path: templates/bgp_core.txt
```

This template definition:

- **name**: A unique identifier for the template
- **type: file**: Specifies this is a file-based template
- **path**: Relative path to the configuration file

## Step 3: Apply the Template to the CORE Switch

Update your `data/devices.nac.yaml` to reference the template for the CORE switch:

```yaml
devices:
  core:
    templates:
      - bgp_core_config
```

## Step 4: Deploy the Configuration

Run Terraform to apply the BGP configuration:

```bash
terraform plan
terraform apply
```

## Step 5: Verify the BGP Session

SSH to the CORE switch and verify the BGP session is established:

```bash
ssh -o KexAlgorithms=+diffie-hellman-group14-sha1 -o HostkeyAlgorithms=+ssh-rsa nac_cisco@198.18.1.21
```

Check BGP neighbor status:

```
show ip bgp summary
```

Expected output showing established session:

```
Neighbor        V    AS MsgRcvd MsgSent   TblVer  InQ OutQ Up/Down  State/PfxRcd
198.18.100.2    4 65002      10      12        1    0    0 00:05:23        0
```

## When to Use 'file' vs 'model' Templates

| Scenario | Recommended Type |
|----------|------------------|
| Standard YAML configurations | `model` |
| Large configuration blocks | `file` |
| CLI-generated configurations | `file` |
| Configurations from external tools | `file` |
| Simple, reusable settings | `model` |
